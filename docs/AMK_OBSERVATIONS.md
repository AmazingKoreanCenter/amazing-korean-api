# AMK_OBSERVATIONS — 관찰 영역 (부채 아님)

> **목적**: 발견했지만 부채로 등재될 자격 (5필드) 미충족 항목 보관.
> **부채 아님**: 본 파일 항목 = 작업 대상 X. 능동 처리 금지.
> **승격 조건**: 5필드 모두 채워지면 → `AMK_DEBTS.md` 로 이동.
> **작성일**: 2026-05-12 (G10 / G12 부채 자격 미성립 결정 후 분리)
>
> **5필드 (부채 등재 게이트)**:
> 1. **WHERE** — 구체 파일/path/함수
> 2. **WHAT** — incident class (회귀 / 보안 / 성능 / 사고 패턴 재발)
> 3. **HOW MUCH** — 측정 가능한 종료 조건 (시나리오 N개 / 수치)
> 4. **WHY** — 안 하면 발생할 구체 incident
> 5. **END** — 충분의 정의 (= 작업 종결 시점)

## 1. G10 — src/ 테스트 부족 (2026-05-04 발견, 2026-05-12 관찰 영역 분리)

### 미충족 필드

| 필드 | 명세 | 상태 |
|------|------|:-:|
| WHERE | (부족) "src/" 만 적힘 — 어떤 path 인지 불명확 | ❌ |
| WHAT | 종류 불명 (회귀 방지? 보안? 사고 차단?) | ❌ |
| HOW MUCH | cover % / 시나리오 수치 없음 | ❌ |
| WHY | "테스트 없으면 사고 가능" 가설 수준 | ❌ |
| END | 종료 조건 없음 → 무한 카테고리 (도메인 N × 함수 M × 분기 K) | ❌ |

### 현재 사실

- 2026-05-04 ~ 2026-05-12 기간 누적 테스트:
  - frontend: 0 → 249 (122 → 249 = 본 세션 +127)
  - backend lib: ~150 → 212
  - integration: 0 → 25
- 추가된 테스트의 **외부 측정 가능한 효과**: 8/12 = incident-based (B3/B4/B7/G1/G2/G2-1/G16/J1-J3 패턴), 4/12 = 메타 (개발자 본인용)
- **G10 자체는 능동 처리 불가** = 부족의 명세가 없음

### 승격 가능 시점

다음 조건 충족 시 부채 카탈로그 (`AMK_DEBTS.md`) 이동 검토:
- 실제 incident 발생 → 해당 path 명시 부채로 신규 entry (예: "결제 webhook 회귀 방지" 같은 구체)
- 사용자가 product critical failure mode 정의 → 그 path 만 cover

### 관련 작업 이력 (참조)

2026-05-04 ~ 2026-05-12 기간 G10 라벨로 처리된 PR:
- `AMK_STATUS.md` entry #59~#138 중 G10 관련 (~30 PR)

## 2. G12 — cargo-geiger 미설치 (2026-05-04 발견)

### 미충족 필드

| 필드 | 명세 | 상태 |
|------|------|:-:|
| WHERE | 도구 미설치. 적용 대상 = 전체 codebase (현재 unsafe 0건) | ⚠️ |
| WHAT | unsafe 코드 감지 자동화 | ✅ |
| HOW MUCH | 종료 조건 없음 ("우선순위 낮음" 으로 무한 보류) | ❌ |
| WHY | 현재 unsafe = 0건 → 감지 대상 없음 → 안 해도 영향 0 | ❌ |
| END | unsafe 도입 시점이 명시 안 됨 (가설 부채) | ❌ |

### 현재 사실

- `AMK_DEBTS.md` 자체 표기: "unsafe 0건이라 우선순위 낮음"
- 즉 도입 직전 까지 0% 작동 도구 = 가설 부채
- 실제 unsafe 코드 추가 시 시점에 동시 도입 검토 = 올바른 트리거 시점

### 승격 가능 시점

- src/ 에 `unsafe` keyword 첫 추가 시 → 부채로 승격 + cargo-geiger 도입 PR

## 3. 응답 헤더 `X-Frame-Options` 중복 (2026-05-17 발견, 2.6 prod 검증 중)

### 미충족 필드

| 필드 | 명세 | 상태 |
|------|------|:-:|
| WHERE | prod 응답에 `X-Frame-Options: DENY`(앱 `security_headers`) + `X-Frame-Options: SAMEORIGIN`(nginx) 둘 다 | ✅ |
| WHAT | 정돈 차원 (보안 incident 아님) | ⚠️ |
| HOW MUCH | 종료 조건 = 헤더 1개로 통일 (단일 시나리오) | ⚠️ |
| WHY | 보안 구멍 아님 — CSP `frame-ancestors 'none'` 가 권위라 프레이밍은 이미 차단. 중복 XFO 충돌 시 브라우저는 보수적(차단) 처리 | ❌ |
| END | "안 고치면 발생할 incident" 없음 = 부채 자격 미성립 | ❌ |

### 현재 사실

- 2.6 CSP 도입 후 prod 헤더 실측에서 발견. 앱 미들웨어 `DENY` + nginx `add_header X-Frame-Options SAMEORIGIN` 동시 출력.
- 실 보안: `Content-Security-Policy: ... frame-ancestors 'none'` 가 modern 브라우저 권위 — 프레이밍 차단은 보장됨. XFO 중복은 무해(레거시 브라우저도 충돌 시 차단 측).
- 부채 아님 = 능동 처리 금지. nginx conf 정리 작업이 별건으로 생기면 그때 1줄 제거(앱 `DENY` 가 더 엄격하므로 nginx 측 제거).

### 승격 가능 시점

- nginx conf 리팩터 PR 이 별건으로 발생 시 inline 정리 (단독 작업 가치 없음).
