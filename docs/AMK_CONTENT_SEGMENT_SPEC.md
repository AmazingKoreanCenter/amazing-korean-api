# AMK 온라인 콘텐츠 세그먼트 포맷 스펙 (번역·DB 적재 계약)

> **목적**: `amazing-korean-books/해설집/*.txt` 세그먼트 원본 → (번역) → api DB 적재의 **공통 계약**.
> **상태**: 🟡 **DRAFT / 잠정** — 온라인 콘텐츠 내용이 **아직 미확정**. 확정 시 **DB 스키마는 대폭(또는 전면) 재설계 가능**(사용자 결정 2026-06-04). 본 문서는 *세그먼트 포맷(입력)* 과 *표 재조립 규칙* 처럼 **DB 모양과 무관하게 재사용되는 부분**을 먼저 고정한다.
> **근거**: `amazing-korean-books/해설집/*.txt` **33개 파일 전수 집계**(2026-06-04). 추측 아님.
> **SSoT 방향**: 콘텐츠의 원본은 이 **세그먼트 .txt**(books). DB는 여기에 맞춰 설계한다(DB 먼저 만들고 콘텐츠를 맞추지 않는다).

---

## 1. 이게 무엇인가
콘텐츠를 **번역 가능한 최소 단위(블록)** 로 쪼개고, 각 블록에 한국어(KO)·영어(EN)를 단 텍스트 파일. **EN이 번역 소스**(→ 다국어, Mac Mini 파이프라인). 이 블록들이 그대로 **DB 적재 단위**가 된다.

- 1 레슨 = 1 파일 (예: `1~10 문장구조 주어 형용사.txt`).
- 파일당 블록 수: 수십~수백 (단순 레슨 242 / 문법 레슨 더 많음).

## 2. 블록 포맷
```
[05_011] [SECTION]              ← [블록ID] [타입(:위치태그)]
KO: 저는 행복합니다.             ← 한국어 원본
EN: 1) I am happy.              ← 영어 (번역 소스)
```
- **블록ID** `NN_MMM`: `NN`=레슨(페이지) prefix, `MMM`=레슨 내 순번. 멱등 재시드 키 후보.
- **위치태그**: `TABLE_HEADER`/`TABLE_CELL` 만 `:Tn_Rr_Cc`(표n·행r·열c) 좌표를 가진다. 나머지 타입은 위치태그 없음.

## 3. 세그먼트 타입 (12종, 전수)
| 타입 | 빈도 | 역할 | 예 |
|------|-----:|------|-----|
| `TITLE` | 33 | 레슨 제목(파일당 1) | 문장구조 1 / Sentence Structure 1 |
| `PARAGRAPH` | 246 | 교육 설명문 / 섹션 라벨 | "한국어에서 가장 기본적인 문장은…" |
| `PATTERN` | 62 | 패턴 다이어그램 조각 | 주어 / 명사+조사 / 서술어 / 형용사(격식체) |
| `SECTION` | 305 | 연습 문장(한국어 문장 + 영어 뜻) | 저는 행복합니다 / 1) I am happy |
| `VOCAB` | 698 | 어휘쌍 | 저=I, 행복하다=am happy |
| `UI` | 967 | 인터페이스 라벨(메타) | 📚 어휘, 📖 해설 |
| `NOTE` | 22 | 강조 규칙/주의 | "⚠️ 구조 3의 핵심 규칙: 목적어가 있으면 서술어는 동사…" |
| `BLOCKQUOTE` | 127 | 문법 활용 예시(인라인) | "가다 → 가 + 려고 합니다 = 가려고 합니다" |
| `OTHER` | 281 | 기타 구조 콘텐츠(문법 레슨의 개념카드 등) | "① 능력 + 가능 / Ability + Possibility" |
| `TABLE_HEADER:Tn_Rr_Cc` | 1594 | 표 헤더 셀 | 영어/English, 역할/Role |
| `TABLE_CELL:Tn_Rr_Cc` | 3796 | 표 데이터 셀 | 저는 / Subject / 저+는 |
| `PRACTICE` | 135 | 활동 라벨(메타) | Read Along, Sentence Matching |

## 4. 특수 마커 (KO/EN 값에 들어가는 `(xxx)`)
각 셀/블록의 **"실내용이 어느 쪽이고, 번역을 타는지"** 를 명시한다. 한 쪽에 실내용이 있으면 **반대쪽에 마커**가 온다.

| 마커 | 빈도 | 의미 | 번역? |
|------|-----:|------|------|
| `(empty)` | 405 | 그 언어 칸 비움 | — |
| `(symbol_only)` | 1058 | 기호·장식만(`+`,`→`,`=`,아이콘) | ❌ (언어 무관) |
| `(ko_learning)` | 386 | 한국어 학습 내용(블록) | ❌ |
| `(content_ko_learning)` | 44 | 한국어 학습 내용(비표 블록 변형) | ❌ |
| `(table_content_ko_learning)` | 1428 | 표 셀이 한국어 학습 내용 → KO에 실내용, EN은 이 마커 | ❌ |
| `(table_content_en)` | 1438 | 표 셀이 영어 콘텐츠 → EN에 실내용, KO는 이 마커 | ✅ EN→다국어 |

> 즉 표 셀은 **열 위치로 추론할 필요 없이** 마커가 직접 "이 셀은 영어(번역)냐 / 한국어 학습(불역)이냐 / 기호냐 / 빔"을 알려준다.
> (`(please)`/`(rain)` 등 극소수는 마커 아닌 실제 콘텐츠 — 파싱 시 화이트리스트로 위 6종만 마커 처리.)

## 5. 표 재조립 규칙 (좌표 → 구조) — **반드시 구현 (사용자 확정)**
`TABLE_HEADER`/`TABLE_CELL` 의 `Tn_Rr_Cc` 로 표를 복원한다.
1. **그룹핑**: `Tn`(표 번호)으로 한 표의 셀을 모은다.
2. **격자 배치**: `Rr`(행)·`Cc`(열) 좌표로 격자에 배치. 빠진 좌표 = 빈 셀.
3. **셀 값 해소**: 각 셀의 KO/EN 중 마커 아닌 쪽이 실내용. 마커로 번역 여부 판정(§4).
   - `(table_content_en)` → EN이 실내용, 다국어 번역 적용.
   - `(table_content_ko_learning)`/`(symbol_only)` → 해당 텍스트 그대로(불역).
   - `(empty)` → 빈 셀.
4. **헤더/바디 구분**: `R0`/`R1` = 헤더(colspan 타이틀·열 제목), `R2+` = 데이터.
5. **표 종류**: 문장별 해설표(Tn, 4열 English/Role/Korean/Explanation) + 부록표(예: 형용사·반의어 T11).

→ 이 로직은 **DB 스키마가 어떻게 바뀌든 필요** (입력 포맷 고정 → 구조 복원).

## 6. 번역 흐름
- **소스 = EN**. Mac Mini 파이프라인이 EN → 다국어(35개) 번역 → `content_translations`(또는 후속 모델)에 언어별 적재. SSoT = `reference_translation_pipeline` / `amazing-korean-ai`.
- **번역 제외**: `(ko_learning)` 류 · `(symbol_only)` · `(empty)`. 한국어 학습 텍스트(예: `저 + 는`, `행복하다 = 행복하 + ㅂ니다`)는 모든 언어 동일.
- **콘텐츠 언어 ≠ UI 언어**: 학습 콘텐츠 표시 언어는 사용자가 고른 앱 UI 언어와 **독립**(학습자가 자기 모국어로 해설을 봄). 우리 시스템에 이미 분리됨(`getContentLang()` ↔ i18next).

## 7. 확정 결정 (2026-06-04, 사용자)
1. **모든 항목 번역** — TXT의 콘텐츠는 (마커 제외) 전부 번역해 화면에 표시. **역할(Subject/Predicate)도 번역 대상**(현재 우리 모델은 lang-invariant로 저장 → 정렬 필요).
2. **표 재조립 로직 = 필수 구현**(§5).
3. **콘텐츠 확정 시 DB 대폭/전면 재설계 가능** — 현재 explanation/study 모델은 **잠정**.
4. **UI/PRACTICE 라벨(📚어휘/Read Along) = 메타 표시** — 학습 내용과 별개. 콘텐츠 번역 vs 화면 UI 번역 배치는 **기회비용 따져 추후 결정**(중복 번역 회피 관점).

## 7.5 학습 UX 설계 원칙 (증거 기반 — 딥리서치 2026-06-04, 인용 검증 20/25 확정)
> 상세·인용·반증 = 메모리 `project_content_ux_research`. 이 원칙이 흐름을 정하고, 흐름이 DB 모델을 정한다.

**검증됨(유지)**: 청킹/분할(Rey 2019 보존 d=0.32·전이 d=0.36·인지부하↓) / 시도-먼저=능동회상(prequestion g=.66·Productive Failure g=0.36 — **단 시도 뒤 정답·설명 필수, 지연≠보류**) / 즉시 피드백 / 분산학습(Kim&Webb 2022 L2 중~대).

**조정(핵심 4)**:
1. **의미 경계 강제 분할** — 단순 Next 버튼만으론 보존·인지부하 이득 소멸(d=0.19 비유의), 시스템이 의미단위로 끊어야 효과(d=0.42). 한 스텝 = 한 문장/한 개념.
2. **새 개념=예제 먼저, 복습=시도 먼저** — 초보자가 복잡한 새 문법 첫 대면 시 시도-먼저는 역효과(worked-example effect; Clark·Kirschner·Sweller 2012; 전문성 쌓이면 역전). 새 패턴/조사 도입=예제→페이딩→시도 / 복습·회상=시도먼저. (예제-문제 페어 + 페이딩)
3. **블록 먼저 → 인터리빙** — 인터리빙은 활용/조사에 유익(Choi&Pan 2024)하나 약초보·새 어휘 즉시 혼합은 해로움(Hwang 2025 한국 RCT). 한 개념 고립 숙달 후 혼합.
4. **"2~3분 최적"은 근거 없음(정직)** — 마이크로러닝 "최적 시간(5–10분)" 주장 검증 전부 기각(블로그/RG급 소스). 짧은 세션 근거 = 분할+분산학습이지 마이크로러닝 메타 아님. 시간 숫자 집착 금지.

**증거 반영 레슨 흐름**: ① 새 문법 = 예제 도입(시도X) → ② 페이딩 연습(빈칸 점증) → ③ 문장별 회상(시도먼저→피드백→정답, 한 스텝=한 문장) → ④ 블록 숙달 후 인터리빙(짝맞추기 등) → ⑤ 며칠 뒤 분산 복습(이전 문항 재등장).

→ **DB 함의**: 모델이 ① 항목별 단계(예제/페이딩/회상) ② 분산 복습 스케줄(언제 재등장) ③ 블록→인터리빙 묶음을 표현해야 함.

**미답(별도 제품 분석 필요)**: Q1 선도 앱(Duolingo·LingoDeer 등) 실제 레슨 흐름·세션길이·게이미피케이션 / Q3 한국어 특화 UX(한글 도입·조사·존댓말·로마자 vs 한글전용) / Q5 게이미피케이션 안티패턴 — 학술 아닌 제품/경쟁 분석 영역, 본 라운드 검증 증거 0.

## 8. 미결 / 콘텐츠 확정 후로 보류
- **DB 스키마·적재 모델 확정**: 12 타입 + 마커를 어떤 테이블/컬럼/JSONB로 받을지. 현재 `explanation_unit/block` 재사용 vs 신설/전면개편 — 콘텐츠 확정 후.
- **신규 콘텐츠 수용**: PATTERN(패턴 다이어그램)·VOCAB(어휘)·OTHER(개념카드)·부록표(반의어) 등 현재 DB에 없는 것.
- **#4 라벨 배치**: content_translations vs UI i18n.
- **블록ID(`NN_MMM`) ↔ unit_idx/study_task_idx 연결키 매핑**.
