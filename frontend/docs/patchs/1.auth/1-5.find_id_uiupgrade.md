# 🚀 Backlog: Find ID System Upgrade Plan (Phase 1-5+)

## 📋 개요
Phase 1-5(MVP)에서 구현한 '아이디 찾기'는 보안 정책(Enumeration Safe)을 준수하는 UI 흐름에 집중했습니다.
다음 단계에서는 **실제 이메일 전송 인프라(SMTP)** 구축과 **봇(Bot) 공격 방지**를 통해 시스템의 실용성과 안정성을 확보해야 합니다.

---

## 1. 이메일 시스템 인프라 (Infrastructure & Delivery)

### 📧 SMTP 서버 연동 및 설정 (Critical)
- **현재**: 로직은 돌지만, 실제 메일을 발송할 SMTP 서버(Mailgun, AWS SES, Gmail SMTP 등)가 연결되지 않음.
- **개선**:
  - 백엔드 `config`에 SMTP 호스트, 포트, 인증 정보를 등록.
  - 개발 환경(Dev)에서는 `MailHog` 같은 가상 SMTP를 사용하여 실제 발송 없이 발송 로그만 확인하는 환경 구축 권장.

### ⚡ 비동기 발송 큐 (Async Task Queue)
- **문제**: SMTP 서버 연결 시 응답 시간이 1~3초 지연될 수 있음. 사용자는 버튼을 누르고 한참 기다려야 함.
- **개선**:
  - API는 요청을 받자마자 **즉시 200 OK** 응답.
  - 실제 발송 작업은 백엔드의 **Background Worker (Redis Queue 등)**로 위임하여 비동기 처리.
  - **효과**: 사용자 경험 향상 및 대량 요청 시 서버 부하 분산.

---

## 2. 콘텐츠 및 템플릿 (Content Strategy)

### 🎨 HTML 이메일 템플릿 적용
- **현재**: 단순 텍스트("회원님의 아이디는 ... 입니다")로 전송될 가능성 높음.
- **개선**:
  - 서비스 로고와 브랜드 컬러가 적용된 **HTML 템플릿** 디자인.
  - 모바일에서도 깨지지 않는 **반응형(Responsive) 이메일** 코딩 적용.
  - **보안 문구 추가**: "본인이 요청하지 않았다면 이 메일을 무시하세요."

---

## 3. 보안 및 어뷰징 방지 (Security & Anti-Abuse)

### 🤖 캡차(Captcha) 도입 (Bot Protection)
- **위협**: 공격자가 스크립트를 짜서 무작위 이메일을 대량으로 입력 -> 서비스가 스팸 메일 발송처로 악용됨(Email Bombing).
- **개선**:
  - **Cloudflare Turnstile** 또는 **Google reCAPTCHA v3** (Invisible) 도입.
  - 사용자는 퍼즐을 맞출 필요 없이, 봇일 경우에만 차단.
- **Frontend**: 폼 제출 전 캡차 토큰 획득 -> API 전송.
- **Backend**: 캡차 토큰 검증 후 비즈니스 로직 수행.

### 🚫 일회용 이메일 차단 (Disposable Email Block)
- **개선**: `temp-mail.org` 등 일회용 이메일 도메인 리스트를 활용하여, 허수 가입자의 계정 찾기 시도를 원천 차단하거나 경고 처리.

---

## 4. 확장 기능 (Feature Expansion)

### 📱 휴대폰 번호로 찾기 (SMS)
- **현황**: 현재는 이메일 기반 찾기만 지원.
- **확장**:
  - 국내 서비스 특화: `CoolSMS`나 `Aligo` API 연동.
  - 사용자가 가입 시 휴대폰 인증을 했다면, SMS로 아이디를 발송해주는 옵션 추가.
